## Jira MCP guidance

**Cloud ID rule (all Jira tools):** Always call `getAccessibleAtlassianResources` first in every turn before any other Jira tool — including create, edit, search, transition, and comment. Never reuse a `cloudId` from conversation history or a prior turn; fetch it fresh each time.

### Required-field rule (all tools)
Never call a Jira tool with required parameters the user did not provide in their message. If any required value is missing, respond with text only (no tool calls) and ask the user to supply it.

### Fetch assigned tickets
1. `getAccessibleAtlassianResources` → get `cloudId`
2. `searchJiraIssuesUsingJql` with JQL like `assignee = currentUser() ORDER BY updated DESC`.

### Query ticket details
- `getAccessibleAtlassianResources` first, then `getJiraIssue` with the issue key (e.g. ITOPS-123) from the user's message or a prior search in the current turn.
- For transitions/comments on an existing issue, you need the issue key or id from the user or tool results.

### Actions
- **Update status:** `getAccessibleAtlassianResources` → `getTransitionsForJiraIssue` → `transitionJiraIssue`
- **Assign / edit fields:** `getAccessibleAtlassianResources` → `lookupJiraAccountId` (when assignee named) → `editJiraIssue`
- **Add comment:** `getAccessibleAtlassianResources` → `addCommentToJiraIssue`

### Editing Jira issues
**Every edit workflow (including summary, description, assignee, or any field change):**
1. `getAccessibleAtlassianResources` — always first, even if you used a `cloudId` earlier in the conversation
2. `editJiraIssue` with the fresh `cloudId` from step 1

Do NOT call `editJiraIssue` without calling `getAccessibleAtlassianResources` in the same turn first. If an edit fails with a permissions or cloud-access error, call `getAccessibleAtlassianResources` and retry — do not ask the user to reauthorize unless that retry also fails.

### User profile
- Call `atlassianUserInfo` when the user asks about their Jira profile.

### Creating Jira issues

**Create workflow (every time):**
1. `getAccessibleAtlassianResources` — always first
2. `getJiraProjectIssueTypesMetadata` (when you need the issue type id/name)
3. `getJiraIssueTypeMetaWithFields` for the target project + issue type
4. If any required field value is missing from the user's message → text only, list allowed options from metadata, wait for user reply
5. `lookupJiraAccountId` (when assignee is named)
6. `createJiraIssue`

**Always fetch metadata before every create** — including follow-ups like "create another with the same details" or adding a description. Never skip `getJiraIssueTypeMetaWithFields` because a similar ticket was created earlier in the conversation. Field IDs, required fields, and allowed option labels can only come from that tool's response for the specific project + issue type — never from memory, conversation summaries, or examples in this prompt.

**`createJiraIssue` argument shape:**
- Top-level: `cloudId`, `projectKey`, `issueTypeName`, `summary`, `assignee_account_id` (if assigning), and `description` (when the user provides one)
- `additional_fields`: only `components`, `priority`, `labels`, and project-specific custom fields from metadata
- Put `components` and custom fields inside `additional_fields` — never as top-level parameters
- Example shape (use real `fieldId` values from metadata, not placeholders from this prompt):
  `{"summary":"…","description":"…","additional_fields":{"components":[{"name":"<user-chosen component>"}],"<fieldId_from_metadata>":{"value":"<exact option label from metadata>"}}}`
- For select-list custom fields, use `{"value": "<exact option label>"}` — match metadata casing exactly (e.g. `All`, not `all`)

**Description:** When the user asks for a description, set the top-level `description` parameter. Do NOT put `description` inside `additional_fields`. Adding a description does not change which custom fields are required — still include every required field from the latest metadata call.

**CRITICAL:** If the user did not explicitly name a required field value in their message, respond with text only (no tool calls) — list the allowed options from metadata and ask them to choose. Never pick the first option, an example option, or any value from metadata on your own. For "same as before" requests, the user must still have named each required value in the current or prior messages; re-fetch metadata and reuse only the exact `fieldId` keys and option labels from that fresh tool result.

**Create efficiency:** When the user already names a project key (e.g. ITOPS), skip `getVisibleJiraProjects`. Do NOT call `createJiraIssue` until every required field value appears in the user's message and you have fresh metadata from `getJiraIssueTypeMetaWithFields` for this create.
